CONCURRENT TASK TREES (CTT) - REFERENCE GUIDE
============================================

CTT is a formal model for representing complex workflows with concurrent, sequential, and choice-based task execution.
It enables precise specification of task relationships, dependencies, and execution semantics.

## CORE CONCEPTS

A CTT tree consists of tasks arranged hierarchically. Each task is an atomic unit of work with:
- Required metadata: unique ID, title, and description
- Optional properties: status, flags (optional/iterative), and child tasks
- Operators define HOW child tasks relate to each other

## TWO WAYS TO STRUCTURE CHILDREN

### METHOD 1: FLAT CHILDREN (Simple, Legacy)
For simple sequential or single-operator structures, use flat list:

  {
    "task_id": "t2",
    "title": "Process Payment",
    "task_description": "Handle payment processing with multiple methods",
    "operator": "choice",
    "children": [
      { "task_id": "t2a", "title": "Credit Card", ... },
      { "task_id": "t2b", "title": "PayPal", ... },
      { "task_id": "t2c", "title": "Bank Transfer", ... }
    ]
  }

### METHOD 2: EXPRESSION TREE (Recommended, W3C CTT Standard)
For complex workflows where operators sit BETWEEN children (like math expressions), use binary operator nodes:

  {
    "task_id": "checkout",
    "title": "Checkout Process",
    "task_description": "Complete purchase with payment and confirmation",
    "children_tree": [
      {
        "operator": "sequence",
        "left": {
          "task_id": "get_details",
          "title": "Get Customer Details",
          "task_description": "..."
        },
        "right": {
          "operator": "choice",
          "left": {
            "task_id": "cc_pay",
            "title": "Process Credit Card",
            "task_description": "..."
          },
          "right": {
            "task_id": "paypal_pay",
            "title": "Process PayPal",
            "task_description": "..."
          }
        }
      }
    ]
  }

VISUAL REPRESENTATION:
Get_Details >> (Process_Card [] Process_PayPal)

In the expression tree above:
- operator "sequence" (>>) sits BETWEEN Get_Details (left) and the choice branch (right)
- operator "choice" ([]) sits BETWEEN the two payment methods

## TASK STRUCTURE

LEAF TASK (0-1 children):
  {
    "task_id": "t1",
    "title": "Authenticate User",
    "task_description": "Validate user credentials against database",
    "status": "pending",
    "optional": false,
    "iterative": false
  }

BRANCH TASK - FLAT STYLE (2+ children with single OPERATOR):
  {
    "task_id": "t2",
    "title": "Process Payment",
    "task_description": "Handle payment processing with multiple methods",
    "operator": "choice",
    "children": [
      { "task_id": "t2a", "title": "Credit Card", ... },
      { "task_id": "t2b", "title": "PayPal", ... }
    ]
  }

BRANCH TASK - EXPRESSION TREE STYLE (binary operators):
  {
    "task_id": "workflow",
    "title": "Complex Workflow",
    "task_description": "Multi-step process with mixed operators",
    "children_tree": [
      {
        "operator": "sequence",
        "left": { ... task or operator node ... },
        "right": { ... task or operator node ... }
      }
    ]
  }

## OPERATORS - EXECUTION SEMANTICS

1. SEQUENCE (>>, enabling, sequential, pipeline)
   - Execute ALL children in STRICT ORDER, one after another
   - Child N cannot start until Child N-1 completes
   - Use when: tasks have dependencies or order matters
   - Example: Fetch Data → Process → Save (must happen in order)
   - Default when 2+ children and no operator specified

2. CHOICE ([], alternative, one-of)
   - Execute EXACTLY ONE of the children
   - Mutually exclusive branches
   - Use when: alternative paths, decision points, error handling
   - Example: Handle Error → (Generate Report) OR (Auto-Retry) OR (Skip)
   - Only one branch executes

3. INTERLEAVING (|, |||, parallel, concurrent)
   - Execute ALL children SIMULTANEOUSLY (in parallel)
   - Order-independent execution
   - Use when: tasks are independent, can run concurrently
   - Example: Run (Unit Tests) AND (Integration Tests) AND (Security Scan) together
   - All must complete before parent finishes

4. DISABLING ([>, interruption, interrupt, override)
   - A task CAN INTERRUPT and override other tasks
   - Used for exception handling, critical events
   - Use when: one task takes priority over others
   - Example: Security Scan finds critical vulnerability → stops other tasks

## EXECUTION RULES

RULE 1: Choose Your Structure
- For simple, single-operator workflows → use "children" + "operator"
- For complex workflows with mixed operators → use "children_tree" with binary operator nodes
- DO NOT mix "children" and "children_tree" in the same task

RULE 2: Operator Placement
- In flat style: only tasks with 2+ children GET an operator
- In expression tree: operators sit between left and right children
- Leaf nodes (0 children) MUST NOT have operator field

RULE 3: Default Behavior
- If 2+ children exist in flat style but no operator → default to SEQUENCE
- Sequential execution is safest, most predictable

RULE 4: Iteration vs Operator
- Do NOT use operator='*' for repetition
- Use iterative=true flag instead
- Example: Unit tests run iteratively over multiple modules

RULE 5: Optional & Iterative
- optional=true: task CAN be skipped
- iterative=true: task REPEATS for multiple items/pages/modules
- Independent flags, can combine

## STATUS VALUES

- pending: not started yet
- running: currently executing
- blocked: waiting for dependency
- done: completed successfully
- failed: execution failed

## PRACTICAL EXAMPLES

EXAMPLE 1: Simple Flat Structure (Web Scraper)
  root_tasks: [
    {
      task_id: "parse",
      title: "Parse Products",
      task_description: "Extract product info from page",
      iterative: true,
      operator: "sequence",
      children: [
        { task_id: "name", title: "Extract Name" },
        { task_id: "price", title: "Extract Price" },
        { task_id: "desc", title: "Extract Description" }
      ]
    }
  ]

EXAMPLE 2: Expression Tree (Complex Workflow)
  root_tasks: [
    {
      task_id: "checkout",
      title: "E-Commerce Checkout",
      task_description: "Complete purchase workflow",
      children_tree: [
        {
          "operator": "sequence",
          "left": {
            "task_id": "cart",
            "title": "Review Cart",
            "task_description": "User reviews items"
          },
          "right": {
            "operator": "sequence",
            "left": {
              "task_id": "shipping",
              "title": "Enter Shipping Address",
              "task_description": "Get delivery info"
            },
            "right": {
              "operator": "choice",
              "left": {
                "task_id": "cc",
                "title": "Credit Card Payment",
                "task_description": "Process credit card"
              },
              "right": {
                "task_id": "paypal",
                "title": "PayPal Payment",
                "task_description": "Process via PayPal"
              }
            }
          }
        }
      ]
    }
  ]

Visualization:
Review_Cart >> (Enter_Shipping >> (Credit_Card [] PayPal))

## VALIDATION CHECKLIST

When creating CTT:
☑ Every task has: task_id (unique), title, task_description
☑ Use EITHER "children" (flat) OR "children_tree" (expression tree), not both
☑ In flat style: only branch nodes (2+ children) have operator field
☑ In expression tree: operator nodes have "operator", "left", "right" fields
☑ All operator values in [sequence, choice, interleaving, disabling]
☑ No null, None, or empty string values
☑ Child task_ids are unique within tree
☑ iterative=true for repeated tasks, not operator='*'
☑ optional=true only for truly skippable tasks
☑ Status values from [pending, running, blocked, done, failed]

## OUTPUT FORMAT

Return ONLY valid JSON:
{
  "root_tasks": [
    { task or operator definition },
    { task or operator definition },
    ...
  ]
}

- No markdown formatting
- No prose or explanations
- No code fences (return raw JSON)
- Ensure valid JSON syntax
- Apply all CTT rules above
